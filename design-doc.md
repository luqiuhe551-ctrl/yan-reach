### 一、系统架构概览


### 二、模块详细设计

#### 模块1：配置模块（用户设定）

**功能**：用户通过对话设定推送规则

**数据存储（config.json）**：

```json
{
  "time_start": 7,
  "time_end": 23,
  "min_count": 1,
  "max_count": 3,
  "push_channel": "telegram",
  "telegram_bot_token": "YOUR_BOT_TOKEN",
  "telegram_chat_id": "YOUR_CHAT_ID",
  "ntfy_topic": "YOUR_TOPIC",
  "ntfy_server": "https://ntfy.sh"
}
```

> **Claude API Key 配置说明**：Claude API Key 建议通过**环境变量** `CLAUDE_API_KEY` 注入，不在 config.json 中明文存储。如确需存储在配置文件中，可增加 `claude_api_key` 字段（不推荐）。

**用户交互示例**：

- 用户："改成全天候模式" → 更新 `time_start: 0, time_end: 24`
- 用户："每天推2-4次" → 更新 `min_count: 2, max_count: 4`
- 用户："换成 ntfy 推送" → 更新 `push_channel: "ntfy"`

**配置说明**：config.json 中同时保留两套渠道凭证，用户切换渠道时仅修改 `push_channel` 字段，另一套凭证保留在配置中不清理（方案A）。

#### 模块2：调度模块（APScheduler 事件驱动 + 持久化）

**功能**：每天0点自动计算今日推送时间点，并注册为一次性定时任务

**执行逻辑**：

```
每日0点触发：
  1. 读取 config.json
  2. 在 time_start ~ time_end 范围内
  3. 随机生成 min_count ~ max_count 个时间点（精确到分钟）
  4. 存入 schedule.json（作为审计日志，记录今日安排了哪些时间点）
  5. 对每个时间点，调用 APScheduler 注册一次性任务：
     scheduler.add_job(推送逻辑, 'date', run_date='YYYY-MM-DD HH:MM:00')
  6. 任务到期自动触发，精确到秒，无需轮询
```

**技术选型**：


| 方案            | 说明                                                                                    |
| :-------------- | :-------------------------------------------------------------------------------------- |
| **APScheduler** | Python 定时任务库，支持`date` 触发器注册一次性任务                                      |
| **Job Store**   | 使用**SQLAlchemyJobStore** + SQLite 持久化，防止程序重启后任务丢失                      |
| **启动恢复**    | `main.py` 启动时检查 `schedule.json` 中今日是否有未执行的推送时间点，若有则重新注册任务 |
| **替代轮询**    | 不再使用轮询检查，消除秒级边界风险，降低资源消耗                                        |

**schedule.json 定位**：`schedule.json` 不再作为调度驱动文件，而是作为**审计日志**，记录当天已安排的推送时间点。APScheduler 注册完任务后不再读取该文件；仅在程序启动恢复时读取。

#### 模块3：内容模块（匹配+融合+生成+容错+去重）

**功能**：时间到达时，融合**时段标签 + 短期记忆 + 长时记忆**，调用AI生成内容

**数据存储**：


| 文件            | 用途                                               | 数据来源              |
| :-------------- | :------------------------------------------------- | :-------------------- |
| `context.json`  | **短期记忆**：存储最近10-20条对话/推送记录         | 本地维护              |
| **Ombre-Brain** | **长时记忆**：情感坐标标记的历史记忆，支持语义检索 | MCP 调用`breath` 工具 |

**执行逻辑**：

```
时间到达（如 08:15）：
  1. 读取当前时间 → 调用 get_time_segment(hour, minute) 判断时段
  2. 读取 context.json → 获取最近10-20条对话摘要（短期记忆）
  3. 调用 Ombre-Brain.breath（超时3秒）→ 检索相关历史记忆
     - 成功：注入长时记忆
     - 超时/失败：降级，跳过该维度，推送照常执行
  4. 构建融合 Prompt（见下方示例）
  5. 调用 Claude API 生成内容
  6. 【去重校验】使用 jieba 分词后计算词级 Jaccard 相似度
     - 对比最近3次推送内容，相似度 > 0.3 则触发重新生成
     - 追加"换一个角度"指令，最多重试 1 次
     - 仍重复则直接推送
  7. 推送完成后：
     a. 更新 context.json（追加本次推送摘要，保留最近20条）
     b. 调用 Ombre-Brain.hold（异步重试，最多3次，不阻塞主流程）
```

**时段判断函数（精确边界）**：

```python
def get_time_segment(hour: int, minute: int) -> str:
    t = hour + minute / 60.0  # 转为小数小时
    if 5.0 <= t < 9.0:   return "清晨"
    if 9.0 <= t < 12.0:  return "上午"
    if 12.0 <= t < 14.0: return "午间"
    if 14.0 <= t < 18.0: return "下午"
    if 18.0 <= t < 21.0: return "晚间"
    return "深夜"  # 21:00-04:59 全部归深夜
```

**内容匹配规则（时段标签）**：


| 时段        | 标签 | Prompt 片段                                |
| :---------- | :--- | :----------------------------------------- |
| 05:00-08:59 | 清晨 | "现在是清晨，结合记忆写一句温暖的早安问候" |
| 09:00-11:59 | 上午 | "上午时段，结合记忆提醒用户适当休息"       |
| 12:00-13:59 | 午间 | "午间时间，结合记忆聊聊午餐或午后小憩"     |
| 14:00-17:59 | 下午 | "下午时段，结合记忆分享轻松话题"           |
| 18:00-20:59 | 晚间 | "晚间时间，结合记忆聊点放松的话题"         |
| 21:00-04:59 | 深夜 | "深夜时段，结合记忆简短安抚，提醒早睡"     |

**容错策略汇总**：


| 场景                | 策略                                          |
| :------------------ | :-------------------------------------------- |
| **breath 调用失败** | 降级：跳过长时记忆，仅用时段+短期记忆生成内容 |
| **hold 调用失败**   | 异步重试（最多3次），不阻塞推送               |
| **超时设置**        | 统一 3 秒超时，超时即降级                     |
| **Claude API 失败** | 记录日志，本次跳过，不影响后续推送            |

**融合 Prompt 示例**：

> "现在是 [时段标签]，北京时间 [当前时间]。
>
> 【短期记忆（最近对话）】：
> [context.json 中提取的最近3-5条对话摘要]
>
> 【长时记忆（Ombre-Brain 检索）】：
> [breath 返回的情感记忆片段]
>
> 请综合以上信息，生成一句自然、亲切的推送内容。语气像朋友闲聊，不要像机器人。**注意：避免与上次推送内容重复。** 如果短期和长期记忆有重复，由你自行判断以哪个为主。"

#### 模块4：发送模块（双通道推送）

**功能**：将AI生成的内容通过用户选择的渠道推送

**支持渠道**：


| 渠道             | 说明                   | 配置字段                                  |
| :--------------- | :--------------------- | :---------------------------------------- |
| **Telegram Bot** | 稳定可靠，保留消息历史 | `telegram_bot_token` + `telegram_chat_id` |
| **ntfy.sh**      | 轻量级推送，支持自建   | `ntfy_topic` + `ntfy_server`              |

**执行逻辑**：

```
收到 content 后：
  1. 读取 config.json 中的 push_channel
  2. 若为 "telegram" → 调用 Telegram Bot API 发送
  3. 若为 "ntfy" → 调用 ntfy.sh API 发送
  4. 记录发送状态日志（成功/失败）
```

### 四、文件清单


| 文件名            | 用途                                                                         |
| :---------------- | :--------------------------------------------------------------------------- |
| `config.json`     | 存储用户配置（时间、次数、推送渠道、凭证）                                   |
| `schedule.json`   | **审计日志**：存储每日随机时间表，记录当日安排了哪些推送（供启动恢复时读取） |
| `context.json`    | **短期记忆库**：存储最近10-20条对话/推送摘要                                 |
| `main.py`         | 主程序（APScheduler 调度 + 内容融合 + 推送控制 + 启动恢复）                  |
| `send.py`         | 推送模块（Telegram + ntfy 双通道）                                           |
| `ombre_client.py` | Ombre-Brain MCP 客户端（breath / hold 调用，含容错）                         |
| `utils.py`        | 工具函数（时段判断、jieba分词去重校验等）                                    |
| `scheduler.db`    | SQLite 数据库（APScheduler SQLAlchemyJobStore 持久化文件，自动生成）         |

### 五、给 Codex 的 Prompt（最终版）

> "请帮我用 Python 写一个主动推送 Skill，部署在云服务器：
>
> 1. 使用 `config.json` 存储配置：推送时间段（`time_start` / `time_end`）、每日推送次数范围（`min_count` / `max_count`）、推送渠道（`push_channel`：telegram 或 ntfy）、以及对应的渠道凭证。两套渠道凭证同时保留，切换时只改 `push_channel` 字段。Claude API Key 通过环境变量 `CLAUDE_API_KEY` 注入。
> 2. 每天0点自动生成当日随机推送时间表（在时间段内随机选取 min~max 个时间点），存入 `schedule.json` 作为审计日志。然后使用 **APScheduler** 的 `date` 触发器为每个时间点注册一次性任务，精确到秒触发，无需轮询。APScheduler 使用 **SQLAlchemyJobStore** + SQLite 持久化（`scheduler.db`）。程序启动时检查 `schedule.json` 中今日是否有未执行的推送时间点，若有则从 SQLite 恢复或重新注册任务。
> 3. 任务触发后，推送时内容由三路信息融合生成：
>    a. 当前时段标签：调用 `get_time_segment(hour, minute)` 函数，用小数小时比较精确判断（清晨/上午/午间/下午/晚间/深夜）
>    b. 短期记忆：读取 `context.json` 中最近10-20条对话摘要
>    c. 长时记忆：通过 MCP 调用 Ombre-Brain 的 `breath` 工具检索情感记忆，设置 **3秒超时**，超时或失败则降级跳过该维度，推送照常执行
> 4. 构建融合 Prompt，明确加入"避免与上次推送内容重复"的指令，然后调用 Claude API 生成最终文本。
> 5. **去重校验**：使用 **jieba** 分词后计算词级 Jaccard 相似度，与 `context.json` 中最近3次推送内容对比。若相似度 > 0.3，触发重新生成（追加"换一个角度"指令），最多重试1次，仍重复则直接推送。
> 6. 根据 `push_channel` 选择发送方式：
>
>    - telegram：调用 Telegram Bot API
>    - ntfy：调用 ntfy.sh API
> 7. 推送完成后：
>    a. 更新 `context.json`，追加本次推送摘要，保留最近20条
>    b. 通过 MCP 调用 Ombre-Brain 的 `hold` 工具存储本次推送记忆，**异步执行，失败自动重试最多3次，不阻塞主流程**
>
> 请提供完整可运行的代码，包含所有模块。"
